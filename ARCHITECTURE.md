# Архитектура проекта SAA-2025 - Полное описание файлов

## Обзор проекта

SAA-2025 - это компилятор/транслятор для собственного языка программирования, который преобразует исходный код в ассемблерный код (MASM). Проект состоит из нескольких модулей, реализующих различные этапы компиляции.

---

## Структура проекта

### 1. Главный файл программы

#### `SAA-2025/SAA-2025.cpp`
**Назначение:** Точка входа в программу, координирует все этапы компиляции.

**Основные функции:**
- Инициализация логов и выходных файлов
- Парсинг параметров командной строки
- Последовательное выполнение этапов:
  1. Лексический анализ (FST::LexAnalyzer)
  2. Синтаксический анализ (MFST::Mfst)
  3. Семантический анализ (Semantic::doAnalyse)
  4. Преобразование в польскую нотацию (PolishNotation::DoPolish)
  5. Генерация кода (CG::Generate)
- Обработка ошибок и логирование

## 3. Модуль ввода

### `SAA-2025/In.h` и `In.cpp`
**Назначение:** Чтение и предобработка исходного файла, разбиение на лексемы.

**Структура IN:**
- `size` - общий размер файла
- `lines` - количество строк
- `ignor` - количество игнорируемых символов
- `lexems` - список лексем (структура lexem с полями line, col, lexem)

**Классификация символов (IN_CODE_TABLE):**
- `T` (1024) - текстовый символ (буквы, цифры)
- `F` (2048) - запрещенный символ
- `L` (256) - разделитель/оператор
- `C` (128) - символ начала/конца символьной константы (')
- `N` (64) - символ новой строки (\n)
- `D` (32) - символ начала/конца строковой константы (")
- `O` (16) - символ начала комментария ($)
- `S` (0) - пробел

**Обработка:**
- Чтение файла посимвольно
- Классификация каждого символа
- Формирование лексем с сохранением позиции (строка, столбец)
- Обработка строковых и символьных констант
- Обработка комментариев (игнорирование до конца строки)

## 7. Лексический анализ

### `SAA-2025/FST.h` и `FST.cpp`
**Назначение:** Лексический анализ исходного кода с использованием конечных автоматов (FST - Finite State Transducer).

**Структуры:**
- `RELATION` - переход в автомате (символ, следующий узел)
- `NODE` - узел автомата (количество переходов, массив переходов)
- `FST` - конечный автомат (строка, позиция, количество состояний, узлы, состояния)

**Ключевые функции:**
- `execute()` - выполнение автомата для распознавания лексемы
- `LexAnalyzer()` - главная функция лексического анализа

**Распознаваемые лексемы:**
- Ключевые слова: `ubyte`, `string`, `char`, `bool`, `declare`, `func`, `main`, `if`, `do`, `while`, `return`, `add`, `out`
- Идентификаторы (LEX_ID = 'i')
- Литералы (LEX_LITERAL = 'l'): числа, строки, символы
- Операторы (LEX_OPERATOR = 'v'): +, -, *, /, %, &, |, :, =
- Разделители: ; , ( ) { } [ ]
- Специальные: LEX_INVERSION ('n') для ~, LEX_BOOL_OPERATOR ('b')

**Библиотечные функции:**
- `GetDate()` - получение текущей даты
- `GetTime()` - получение текущего времени
- `ord()` - преобразование символа в число
- `chr()` - преобразование числа в символ

**Процесс:**
1. Для каждой лексемы из In создается FST
2. Автомат проверяет соответствие строки шаблону
3. При успехе лексема добавляется в LexTable
4. Идентификаторы и литералы добавляются в IdTable

## 10. Грамматика (Greibach Normal Form)

### `SAA-2025/GRB.h` и `GRB.cpp`
**Назначение:** Определение грамматики языка в нормальной форме Грейбах.

**Структуры:**
- `Rule::Chain` - цепочка правил (размер, массив символов)
- `Rule` - правило грамматики (нетерминал, код ошибки, размер, массив цепочек)
- `Greibach` - грамматика (размер, стартовый нетерминал, дно стека, массив правил)

**Алфавит:**
- Терминалы (TS) - положительные числа, соответствуют кодам лексем
- Нетерминалы (NS) - отрицательные числа

**Макросы:**
- `TS(c)` - создание терминала
- `NS(n)` - создание нетерминала
- `ISNS(s)` - проверка, является ли символ нетерминалом

**Правила грамматики:**
- `S` - стартовый символ (программа)
- `N` - нетерминал для объявлений и операторов
- `E` - выражение
- `M` - продолжение выражения (операторы)
- `F` - параметры функции
- `W` - аргументы функции
- `B` - булево выражение

**Примеры правил:**
```
S -> m { N r E ; }        // main с объявлениями и return
S -> m { r E ; }          // main без объявлений
E -> i                     // идентификатор
E -> l                     // литерал
E -> ( E )                 // выражение в скобках
E -> i ( W )               // вызов функции
E -> E M                   // выражение с оператором
M -> v E                   // оператор и выражение
```

**Функции:**
- `getGreibach()` - получение грамматики
- `getRule()` - получение правила по нетерминалу или индексу
- `getNextChain()` - получение следующей цепочки правила

## 11. Синтаксический анализ (MFST)

### `SAA-2025/MFST.h` и `MFST.cpp`
**Назначение:** Синтаксический анализ методом магазинного автомата (MFST - Machine Finite State Transducer).

**Структуры:**
- `MfstState` - состояние автомата (позиция на ленте, номер правила, номер цепочки, стек)
- `MfstDiagnosis` - диагностика ошибки (позиция, код шага, номер правила, номер цепочки)
- `Mfst` - магазинный автомат

**Поля Mfst:**
- `lenta[]` - входная лента (последовательность терминалов)
- `lenta_position` - текущая позиция на ленте
- `lenta_size` - размер ленты
- `st` - стек автомата
- `greibach` - грамматика
- `lex` - лексические таблицы
- `storestate` - стек состояний для бэктрекинга
- `deducation` - вывод (последовательность примененных правил)

**Коды возврата (RC_STEP):**
- `NS_OK` - успешное применение правила
- `NS_NORULE` - нет правила для нетерминала
- `NS_NORULECHAIN` - нет подходящей цепочки правила
- `NS_ERROR` - ошибка
- `TS_OK` - успешное совпадение терминала
- `TS_NOK` - несовпадение терминала
- `LENTA_END` - конец ленты
- `SURPRISE` - неожиданная ситуация

**Алгоритм:**
1. Инициализация: на стек кладется дно стека и стартовый нетерминал
2. Пока не достигнут конец ленты:
   - Если на вершине стека нетерминал:
     - Ищется правило для этого нетерминала
     - Ищется цепочка правила, начинающаяся с текущего терминала
     - Сохраняется состояние (бэктрекинг)
     - Нетерминал заменяется цепочкой (в обратном порядке)
   - Если на вершине стека терминал:
     - Если совпадает с текущим символом ленты - сдвиг
     - Иначе - восстановление состояния (бэктрекинг)
3. Успех, если стек пуст и достигнут конец ленты

**Функции:**
- `step()` - выполнение одного шага анализа
- `start()` - запуск анализа
- `savestate()` - сохранение состояния
- `reststate()` - восстановление состояния
- `push_chain()` - добавление цепочки на стек
- `savededucation()` - сохранение вывода (последовательности правил)
- `printrules()` - вывод примененных правил

**Трассировка:**
При включенном флаге `showMfst` выводится подробная информация о каждом шаге:
- Номер шага
- Примененное правило
- Текущая лента
- Состояние стека

## 12. Семантический анализ

**Функции:**

**1. `doAnalyse()`** - главная функция, вызывает все проверки:
- `operands()` - проверка операндов

**2. `operands()`** - проверка типов операндов:

## 13. Польская нотация

### `SAA-2025/PolishNotation.h` и `PolishNotation.cpp`
**Назначение:** Преобразование выражений в обратную польскую нотацию (RPN).

**Алгоритм (алгоритм Дейкстры):**
1. Проход по выражению слева направо
2. Если операнд - добавляется в выходную строку
3. Если оператор:
   - Пока на вершине стека оператор с большим или равным приоритетом - извлекается в выход
   - Текущий оператор кладется на стек
4. Если `(` - кладется на стек
5. Если `)` - извлекаются операторы до `(`
6. В конце извлекаются все операторы из стека

**Приоритеты операторов:**
- `(` `)` - 0
- `,` - 1
- `+` `-` - 2
- `*` `/` `%` `\` `:` `&` `|` - 3
- `~` - 4

**Специальная обработка:**
- Функции помечаются символом `@` в стеке
- Параметры функций разделяются запятыми
- После преобразования операторы помечаются как `LEX_OPERATOR`

**Функции:**
- `PolishNotation()` - преобразование одного выражения
- `DoPolish()` - преобразование всех выражений в программе
- `get_priority()` - получение приоритета оператора
- `fixIt()` - исправление лексической таблицы после преобразования

## 14. Генерация кода

### `SAA-2025/CodeGeneration.h` и `CodeGeneration.cpp`
**Назначение:** Генерация ассемблерного кода (MASM) из промежуточного представления.

**Структура генерируемого кода:**

**1. Заголовок (`Head()`):**
- Директивы MASM
- Подключение библиотек
- Объявление внешних функций

**2. Константы (`Constants()`):**
- Строковые константы
- Символьные константы

**3. Данные (`Data()`):**
- Объявление переменных
- Инициализация переменных

**4. Функции (`WriteFunctions()`):**
- Генерация кода для пользовательских функций
- Обработка параметров
- Генерация тела функции
- Обработка `return`

**5. Код (`Code()`):**
- Генерация кода для `main`
- Обработка операторов
- Обработка условий (`if`, `while`, `do-while`)
- Обработка вызовов функций
- Обработка вывода (`out`)

**Генерация выражений (`InvokeExpressions()`):**
- Использование стека для вычисления выражений в RPN
- Генерация команд:
  - `push` - поместить операнд на стек
  - `pop` - извлечь операнд из стека
  - Арифметические операции: `add`, `sub`, `mul`, `div`
  - Битовые операции: `and`, `or`, `not`
  - Операция деления `:` генерируется как `div`

**Обработка типов:**
- `UBYTE` - работа через `eax`/`al`
- `BOOL` - работа через `eax`/`al`
- `CHR` - работа через `eax`/`al` с `movzx`
- `STR` - работа через указатели (`offset`)

**Вызов функций:**
- Внутренние функции: `invoke $имя_функции, параметры`
- Внешние функции: `invoke имя_функции, параметры`
- Результат функции помещается на стек

**Условия:**
- Генерация меток для переходов
- Использование `cmp` и условных переходов (`je`, `jne`, и т.д.)

**Циклы:**
- Генерация меток начала и конца цикла
- Условные переходы для `while` и `do-while`

## Пример работы

**Входной файл (in.txt):**
```
main
{
    declare ubyte a;
    a = 5 + 3;
    out a;
    return 0;
}
```

**Процесс:**
1. Лексический анализ создает лексемы: `main`, `{`, `declare`, `ubyte`, `a`, `;`, `a`, `=`, `5`, `+`, `3`, `;`, `out`, `a`, `;`, `return`, `0`, `;`, `}`
2. Синтаксический анализ проверяет соответствие грамматике
3. Семантический анализ проверяет типы
4. Польская нотация: `5 3 +` вместо `5 + 3`
5. Генерация кода создает MASM код с командами `push 5`, `push 3`, `add eax, ebx`

## Заключение

Проект SAA-2025 представляет собой полнофункциональный компилятор с классической архитектурой:
- Лексический анализ (FST)
- Синтаксический анализ (MFST)
- Семантический анализ
- Генерация кода

Все модули тесно связаны и работают с общими структурами данных (LexTable, IdTable), что обеспечивает целостность процесса компиляции.
